# Company/Pipeline Domain

This README is a **non-authoritative migration entrypoint**. Current authority
remains at the canonical repository paths linked below; temporary compatibility
entrypoints remain at the legacy paths until a later migration module removes
them.

## Current ownership

- Company/Pipeline scope and contract ownership are canonical in the
  [Data Protocol](./docs/README.md).
- Research execution is canonical in the
  [Company/Pipeline Research Workflow](./docs/research-workflow.md).
- Company/Pipeline types, loaders, filters, portfolio logic, constants, the
  selector read model, and the `asset-alias-types.json` contract vocabulary are
  canonical under `domains/company-pipeline/lib/`. The remaining `lib/programs/`
  paths are temporary compatibility shims only (pending Module 9).
- `asset-alias-types.json` moved from `lib/programs/` to its canonical home at
  `domains/company-pipeline/lib/asset-alias-types.json` beside its consumers
  (Module 8, D3); the `scripts/data-registry.mjs` validator and
  `constants.ts` read it there.
- Editable Company/Pipeline data, the controlled vocabularies, and the synthetic
  fixtures are Company/Pipeline-owned (Module 7, D4) and were **physically
  relocated** into the domain in the post-Module-8 D4 execution module:
  `domains/company-pipeline/data/companies/`,
  `domains/company-pipeline/data/registries/`, and
  `domains/company-pipeline/data/validation-fixtures/synthetic/`. The
  `scripts/data-registry.mjs` path constants and the `constants.ts` registry
  import point at these paths; the validator context labels, generated bytes,
  ordering, and validation behavior are unchanged.
- The Company/Pipeline generated aggregates
  (`data/generated/{companies,pipeline-programs,regimens}.json`) are
  Company/Pipeline-owned but **remain in the shared `data/generated/` sink at the
  repository root**; the D4 execution retained that sink in place (mirroring the
  Module 8 retention of `scripts/data-registry.mjs`) rather than splitting it.

## Intended future ownership

Later migration modules may place remaining settled Company/Pipeline code,
editable data, fixtures, and domain-owned generated artifacts under this root.
Each relocation requires its own approved module and validation boundary.

## Migration status

Module 1 created this entrypoint. Module 2 resolved D2: `ComponentReference`
remains Company/Pipeline-owned, while `RecordMetadata` and `SourceReference`
are shared provenance types. Module 3 moved the settled authoritative
documentation and the types, loaders, filters, portfolio logic, and constants
while preserving legacy import and documentation entrypoints. Module 5 resolved
D5 for Company/Pipeline: the selector read model moved into
`domains/company-pipeline/lib/` byte-identically behind a compatibility shim,
with no data, validator, generator, or generated-output change. Module 6
resolved D6 (Application/UI phase): `app/` is root-pinned for the current
Next.js architecture and this migration program, not relocatable under a
domain root. Module 7 decision-resolved D4: it assigned semantic domain
ownership of source data, registries, fixtures, and generated outputs (see
Current ownership) but moved no file, and deferred physical relocation to a
post-Module-8 D4 execution module. Module 8 resolved D3: `asset-alias-types.json`
moved to its canonical Company/Pipeline home, and `scripts/data-registry.mjs` is
retained as shared root infrastructure — the single generator/validator
entrypoint — rather than split, relocated, or wrapped. No `data/` file moved and
no generated byte, ordering, validation rule, or command name changed. The
post-Module-8 D4 execution module then physically relocated the
Company/Pipeline-owned editable data (source records, registries, and synthetic
fixtures) into `domains/company-pipeline/data/`, repointing the validator path
constants and the `constants.ts` registry import; the shared `data/generated/`
sink was retained at the repository root, and no generated byte, ordering,
validation rule or message, or command name changed.
